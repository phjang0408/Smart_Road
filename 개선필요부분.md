### 1. DataManager 완성 및 저장 검증

**어떤 문제인지**
DataManager.cs가 생성되어 있지만 CSV/JSON 저장 기능이 완전하지 않고, AutoSaveDay() 반환값의 검증이 없음.

**왜 문제인가?**
현재 MainWindow에서 하루 시뮬레이션이 완료되면 _dataManager.AutoSaveDay()를 호출하는데, 반환값(bool)을 받기만 하고 실패 원인을 파악하지 못함. 파일 시스템 에러, 경로 오류 등이 발생해도 사용자는 "저장 실패"라는 메시지만 봄.

**어디 부분인지**
- 파일: `DataManager.cs` 전체, `MainWindow.xaml.cs` 라인 661, 674-678
- 메서드: `AutoSaveDay()`, `OnDayCompleted()`
- 코드:
  ```csharp
  bool isSaved = _dataManager.AutoSaveDay();
  // 단순 bool만 반환, 상세 에러 정보 없음
  ```

**개선을 어떻게 하면 좋을지**
- DataManager에 상세한 로깅 추가 (성공/실패 이유, 파일 경로)
- SaveResult 클래스 도입 (Status, Message, FilePath 포함)
- CSV 헤더가 SensorData와 일치하는지 검증
- 예외 처리 강화 (IOException, UnauthorizedAccessException 등 세부 처리)

### 2. 역주행 버튼 기능 수정

**어떤 문제인지**
"역주행 시뮬레이션 발생" 버튼이 명세에서는 역주행을 발생시키도록 되어 있지만, 실제로는 긴급 정지(Emergency Stop)만 작동하고 데이터상에 역주행을 발생시키지 않음.

**왜 문제인가?**
버튼의 이름과 실제 동작이 불일치한다. 사용자는 "역주행을 시뮬레이션"하려고 버튼을 누르지만, 실제로는 "긴급 정지만" 발동된다. DataGenerator에서 역주행은 랜덤(2% 확률)으로만 발생하고, UI 버튼에서는 강제로 발생시킬 방법이 없다. 테스트 시 역주행 상황을 의도적으로 만들 수 없어 테스트 커버리지가 낮다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 637, `DataGenerator.cs` 라인 379-382
- 메서드: `BtnTriggerWrongWay_Click()`, `GenerateWrongWay()`
- 코드:
  ```csharp
  // MainWindow.xaml.cs
  private void BtnTriggerWrongWay_Click(object sender, RoutedEventArgs e)
  {
      TriggerEmergencyStop();  // ← 긴급 정지만 작동
  }

  // DataGenerator.cs에서 역주행은 랜덤만 가능
  private bool GenerateWrongWay()
  {
      return _rng.Next(0, 100) < WRONG_WAY_PROBABILITY;  // ← 2% 고정
  }
  ```

**개선을 어떻게 하면 좋을지**
- DataGenerator에 공개 메서드 추가:
  ```csharp
  public void ForceWrongWay(bool enable)
  {
      _forceWrongWay = enable;
  }
  ```
- GenerateWrongWay() 수정:
  ```csharp
  private bool GenerateWrongWay()
  {
      if (_forceWrongWay) return true;
      return _rng.Next(0, 100) < WRONG_WAY_PROBABILITY;
  }
  ```
- MainWindow의 버튼에서 강제 역주행 호출:
  ```csharp
  private void BtnTriggerWrongWay_Click(object sender, RoutedEventArgs e)
  {
      _dataGenerator.ForceWrongWay(true);  // 다음 Tick에 역주행 발생
  }
  ```
- 버튼 텍스트도 명확하게 수정

### 3. 타이머 동기화 (수정됨)
## 배속 시 카운트다운 동기화 문제

**어떤 문제인지**
시뮬레이션 배속을 올려도 신호등 카운트다운 속도가 변하지 않음.

**왜 문제인가?**
`TrafficController`가 `DateTime.Now`(현실 시간)를 기준으로 초를 계산하기 때문임. 배속(2x, 4x)이 적용되어도 신호등은 현실 시간 1배속으로 동작함.

**어디 부분인지**

- 파일: `TrafficController.cs` 라인 176-180 (신호 전환 시간 계산 로직)

**개선을 어떻게 하면 좋을지**
현실 시간 의존성을 제거하고, UI 물리 엔진에서 전달하는 deltaTime(배속이 반영된 흐른 시간)을 받아 카운트다운을 감소시키도록 수정 필요.

```csharp
public void PassTime(double deltaTime)
{
    // 데이터 수신 여부와 상관없이 매 프레임 시간 차감
    _currentPhaseTotalTime -= deltaTime;
}
```

## 데이터 동기화 시간 문제

**어떤 문제인지**
신호 전환 시점에서 "신호 전환 대기 중..."상태가 너무 길게 지속됨

**왜 문제인가?**
UI는 30FPS로 즉시 0초에 도달하지만, `TrafficController`는 `DataGenerator`로부터 다음 센서 데이터(Tick)를 수신받을 때까지 신호 전환 연산을 수행하지 않음.
즉, 데이터 갱신 주기(Tick)와 UI 렌더링 주기 간의 비동기성으로 인해 발생하는 대기 시간임.

**어디 부분인지**

- 파일: `TrafficController.cs` 라인 88-92, 110-115 (`OnSensorDataUpdated`, `ExecuteSignalControl`)

**개선을 어떻게 하면 좋을지**
`TrafficController`가 외부 센서 데이터를 기다리지 않고, UI에서 전달받는 PassTime(deltaTime)을 사용하여 독자적으로 남은 시간을 감소시키고 상태를 변경하도록 로직 분리 필요.

```csharp
public void PassTime(double deltaTime)
{
    _currentPhaseTotalTime -= deltaTime;

    // 시간이 0 이하가 되면 외부 데이터 수신을 기다리지 않고 즉시 신호 전환 로직 실행
    if (_currentPhaseTotalTime <= 0)
    {
        ExecuteSignalControl();
    }
}
```

### 4. CSV 저장 시 타임스탬프 오류

**어떤 문제인지**
SaveToCsv() 메서드에서 DateTime.Now를 저장 시점에 호출하므로 모든 행의 타임스탬프가 동일하다.

**왜 문제인가?**
24시간 시뮬레이션 데이터가 모두 같은 시각(저장 시점)으로 기록되어 데이터 분석이 불가능하다. 데이터의 시계열 정보가 완전히 손실되며 나중에 시뮬레이션 시점을 파악할 수 없다.

**어디 부분인지**
- 파일: `DataManger.cs` 라인 65
- 메서드: `SaveToCsv()`
- 코드:
  ```csharp
  writer.WriteLine($"{DateTime.Now:yyyy-MM-dd HH:mm:ss},...");  // 저장 시점 사용
  ```

**개선을 어떻게 하면 좋을지**
- TrafficUpdateEventArgs에 Timestamp 프로퍼티 추가하여 기록 시점의 시각 저장
- 또는 SensorData에 SimHour(시뮬레이션 시각) 정보 포함
- 저장 시에는 기록된 타임스탬프 사용:
  ```csharp
  writer.WriteLine($"{item.Timestamp:yyyy-MM-dd HH:mm:ss},...");
  ```

### 5. 신호 제어 로직 이중화

**어떤 문제인지**
TrafficController와 MainWindow가 독립적으로 신호 상태를 관리하여 이중 제어가 발생한다.

**왜 문제인가?**
TrafficController는 Emergency(역주행 시 전방향 적색), Congested(혼잡 시 녹색 연장), EnvironmentalHazard(결빙 시 황색 연장) 등 상태별로 신호를 제어한다. 하지만 MainWindow는 TrafficUpdated 이벤트의 신호 상태를 무시하고 자체 _signalTimer와 _isNSGreen 변수로 독립적인 신호 주기를 운영한다. 결과적으로 TrafficController의 핵심 로직(긴급 상황 대응, 교통 혼잡도 반영)이 UI에 전혀 반영되지 않는다.

**어디 부분인지**
- 파일: `TrafficController.cs` (신호 제어), `MainWindow.xaml.cs` 라인 434-461 (독립적 신호 타이머)
- 메서드: `ExecuteSignalControl()` vs `RenderTimer_Tick()` 신호 관리 부분

**개선을 어떻게 하면 좋을지**
- MainWindow의 자체 신호 타이머(_signalTimer, _isNSGreen) 제거
- TrafficUpdated 이벤트에서 받은 Light_N/S/E/W의 Color와 RemainingTime을 UI에 직접 반영
- 신호 표시는 TrafficController의 판단을 따름:
  ```csharp
  _trafficController.TrafficUpdated += (s, e) =>
  {
      this.Dispatcher.Invoke(() =>
      {
          UpdateSignalLights(e.Light_N, e.Light_S, e.Light_E, e.Light_W);
      });
  };
  ```

### 6. TrafficLight 참조 공유로 데이터 오염

**어떤 문제인지**
TrafficUpdateEventArgs에서 TrafficLight 참조를 공유하므로 이후 TrafficController의 상태 변경이 저장된 과거 데이터에도 반영된다.

**왜 문제인가?**
TrafficLight는 클래스(참조 타입)이므로 DataManager._history 리스트의 모든 TrafficUpdateEventArgs가 동일한 Light_N 객체를 참조한다. TrafficController가 Light_N.Color를 변경하면 _history의 모든 항목(과거 24시간 데이터)의 Light_N.Color도 현재 값으로 변경된다. CSV 저장 시 모든 행의 신호가 동일하게 기록된다.

**어디 부분인지**
- 파일: `TrafficController.cs` 라인 88-92
- 메서드: `OnSensorDataUpdated()`
- 코드:
  ```csharp
  Light_N = this.Light_N,  // 참조 공유 (복사 아님)
  Light_S = this.Light_S,
  ```

**개선을 어떻게 하면 좋을지**
- 방법 1: TrafficLight를 record 타입으로 변경 (자동 값 기반 복사)
  ```csharp
  public record TrafficLight(LightColor Color, int RemainingTime);
  ```
- 방법 2: TrafficLight 복사 메서드 추가
  ```csharp
  public TrafficLight Clone() => new TrafficLight
  { Color = this.Color, RemainingTime = this.RemainingTime };

  Light_N = this.Light_N.Clone(),
  ```

### 7. IDisposable 미준수 및 메모리 누수

**어떤 문제인지**
MainWindow에서 리소스 정리가 불완전하고 이벤트 핸들러가 해제되지 않는다.

**왜 문제인가?**
_dataGenerator는 IDisposable을 구현하지만 Stop()만 호출되고 Dispose()는 호출되지 않는다. TrafficController는 DataGenerator 이벤트를 구독한 후 해제 수단이 없다. MainWindow의 익명 람다 이벤트 핸들러(라인 153, 172)가 정리되지 않는다. _renderTimer도 명시적 정리가 누락된다. 장기 실행 시 메모리 누수와 예측 불가능한 동작이 발생할 수 있다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 686 (Window_Closing)
- 파일: `TrafficController.cs` 라인 49-57 (생성자, 구독 해제 불가)
- 파일: `DataGenerator.cs` 라인 101-104 (Dispose 미완성)

**개선을 어떻게 하면 좋을지**
- MainWindow의 Window_Closing에서 명시적 정리:
  ```csharp
  private void Window_Closing(object sender, System.ComponentModel.CancelEventArgs e)
  {
      _renderTimer?.Stop();
      _renderTimer = null;
      _dataGenerator?.Dispose();
      _dataManager?.ClearHistory();
  }
  ```
- TrafficController에 Dispose 구현:
  ```csharp
  private DataGenerator _generator;

  public TrafficController(DataGenerator generator)
  {
      _generator = generator;
      _generator.SensorDataUpdated += OnSensorDataUpdated;
  }

  public void Dispose()
  {
      _generator.SensorDataUpdated -= OnSensorDataUpdated;
  }
  ```

## 개선하면 좋겠는 부분

### 8. 계층 분리

**어떤 문제인지**
UI(MainWindow)가 도메인 로직(TrafficController, DataManager)을 직접 처리하고 있어서 UI와 비즈니스 로직이 강하게 결합되어 있음.

**왜 문제인가?**
Score Logic이나 다른 모듈에서 신호 제어 로직을 건드려야 할 때 UI 코드를 거쳐야 하므로 변수명과 구조가 꼬일 가능성이 높다. 여기서 더 복잡해지면 유지보수가 불가능해진다. 또한 TrafficController의 정책 변경이나 성능 최적화를 테스트할 수 없으므로 버그 발견이 어렵다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 153-169
- 메서드: `InitializeDataGenerator()`의 TrafficUpdated 이벤트 핸들러
- 코드:
  ```csharp
  _trafficController.TrafficUpdated += (s, e) =>
  {
      this.Dispatcher.Invoke(() =>
      {
          _dataManager.RecordData(e);           // ← UI에서 데이터 저장
          ApplyTrafficPolicy(...);              // ← UI에서 정책 적용
      });
  };
  ```

**개선을 어떻게 하면 좋을지**
- 중간 계층 클래스 `TrafficPolicyManager` 추가
- 이 클래스가 TrafficController 이벤트를 받아서 DataManager와 UI에 필요한 데이터만 전달
- 구조:
  ```
  TrafficController → TrafficPolicyManager → MainWindow (UI만 담당)
                   ↓
                DataManager (저장 담당)
  ```
- UI는 렌더링만, 로직은 별도로 테스트 가능

### 9. 렌더링 성능 저하

**어떤 문제인지**
RenderTimer_Tick에서 40개 차량 × 모든 차량을 검사하는 O(N²) 충돌 감지 알고리즘을 사용하고 있어서 매 프레임마다 불필요한 거리 계산이 반복됨.

**왜 문제인가?**
현재 RenderTimer는 30FPS (33ms마다 Tick)로 설정되어 있는데, 40개 차량에서 O(N²) 검사를 30ms 안에 처리하려면 각 Tick마다 1,600번의 거리 계산이 발생한다. 현재는 견딜 만하지만, 차량이 추가되거나 복잡한 로직이 추가되면 프레임이 드롭되어 UI가 버벅거린다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 502-509
- 메서드: `RenderTimer_Tick()`의 앞차 거리 조절 로직
- 코드:
  ```csharp
  for (int i = 0; i < MAX_CARS; i++)
  {
      for (int j = 0; j < MAX_CARS; j++)      // ← O(N²) 검사
      {
          if (car.Direction == "N" && other.Direction == "N" && other.Y - car.Y < 25)
          { canMove = false; break; }          // ← 거리 계산 반복
      }
  }
  ```

**개선을 어떻게 하면 좋을지**
- 각 방향별로 차량 리스트 분리:
  ```csharp
  private List<CarUI> _carsN = new List<CarUI>();
  private List<CarUI> _carsS = new List<CarUI>();
  private List<CarUI> _carsE = new List<CarUI>();
  private List<CarUI> _carsW = new List<CarUI>();

  // 검사할 때 같은 방향 차량끼리만 비교
  for (int i = 0; i < _carsN.Count; i++)
  {
      for (int j = i + 1; j < _carsN.Count; j++)  // ← O(n²)이지만 n은 훨씬 작음
      {
          // 거리 계산
      }
  }
  ```
- 또는 한 번 정렬 후 인접한 것만 체크

### 10. 예외 처리 강화

**어떤 문제인지**
DataGenerator의 UpdateSensorData에서 예외가 발생하면 Debug.WriteLine에만 출력되고, 사용자에게 어떤 오류가 발생했는지 전혀 알려주지 않음.

**왜 문제인가?**
현재 예외는 콘솔에만 출력되므로, 배포된 애플리케이션에서는 오류 정보가 사라진다. 사용자나 개발팀이 문제를 파악할 수 없어 유지보수 어려움. 또한 DataGenerator, TrafficController, MainWindow 각각에서 다른 방식으로 예외를 처리하고 있어 일관성이 없다.

**어디 부분인지**
- 파일: `DataGenerator.cs` 라인 200-204, `MainWindow.xaml.cs` 여러 곳
- 메서드: `UpdateSensorData()`
- 코드:
  ```csharp
  catch (Exception ex)
  {
      System.Diagnostics.Debug.WriteLine($"[DataGenerator] Error: {ex.Message}");
  }
  ```

**개선을 어떻게 하면 좋을지**
- 로깅 시스템 도입 (파일, 이벤트 로그 등)
- 사용자 친화적인 에러 메시지 추가
- 중요 예외(파일 저장 실패, 타이머 오류)는 UI에 표시
- 구조:
  ```csharp
  public class Logger
  {
      public static void Log(string message, LogLevel level) { /* 파일 저장 */ }
  }

  catch (Exception ex)
  {
      Logger.Log($"[DataGenerator] Error: {ex.Message}", LogLevel.Error);
      MessageBox.Show("데이터 생성 중 오류가 발생했습니다.");
  }
  ```

### 11. 단위 테스트 추가

**어떤 문제인지**
DataGenerator, TrafficController 등 핵심 모듈이 단위 테스트 없이 운영되고 있음.

**왜 문제인가?**
현재 버그가 발생해도 전체 UI를 실행해야 테스트 가능함. 각 계산 로직(점수, 노면 상태, 신호)을 독립적으로 검증할 수 없어, 의도하지 않은 부작용을 놓치기 쉽다. 팀원이 다른 모듈을 개발할 때 기존 코드를 깨뜨렸는지 확인할 방법이 없다.

**어디 부분인지**
- 파일: `DataGenerator.cs`, `TrafficController.cs` 등 핵심 로직
- 현재 테스트 프로젝트: 없음

**개선을 어떻게 하면 좋을지**
- NUnit 또는 xUnit 테스트 프로젝트 추가
- 테스트 작성:
  ```csharp
  [TestFixture]
  public class DataGeneratorTests

      [Test]
      public void GetRoadCondition_Freezing_WhenTempBelowZero()
      {
          var gen = new DataGenerator();
          var result = gen.GetRoadCondition(-5, 0);
          Assert.AreEqual("결빙", result);
      }
  }
  ```
- CI/CD 파이프라인에 테스트 자동 실행 추가

### 12. RoadCondition 문자열 하드코딩

**어떤 문제인지**
"결빙", "적설", "습윤", "건조" 등의 문자열이 3개 파일에 걸쳐 하드코딩되어 있다.

**왜 문제인가?**
문자열 리터럴로 비교하므로 오타가 있어도 컴파일 에러가 발생하지 않고 논리 오류만 생긴다. DataGenerator, TrafficController, MainWindow에서 서로 다른 방식으로 비교할 수 있으며, 나중에 "결빙" 대신 "freezing"으로 변경하려면 3개 파일을 모두 수정해야 한다.

**어디 부분인지**
- 파일: `DataGenerator.cs` 라인 314-327, `TrafficController.cs` 라인 117-118, `MainWindow.xaml.cs` 라인 245-251
- 메서드: `GetRoadCondition()`, `CalculateRoadSafetyScore()`, `UpdateUI()`

**개선을 어떻게 하면 좋을지**
- enum 정의:
  ```csharp
  public enum RoadConditionType
  {
      Dry,      // 건조
      Wet,      // 습윤
      Frozen,   // 결빙
      Snowy     // 적설
  }
  ```
- SensorData.RoadCondition 타입 변경 (팀 협의 필요):
  ```csharp
  public RoadConditionType RoadCondition { get; init; }
  ```
- 비교 로직 단순화:
  ```csharp
  if (data.RoadCondition == RoadConditionType.Frozen) { ... }
  ```

### 13. SensorData에 시뮬레이션 시각 정보 누락

**어떤 문제인지**
SensorData에 시뮬레이션 시각(SimHour) 정보가 포함되지 않아 어느 시간대의 데이터인지 알 수 없다.

**왜 문제인가?**
_simHour는 DataGenerator 내부에만 존재하고 생성된 SensorData에는 포함되지 않는다. 저장된 CSV에서 기온이 35도인 데이터를 봐도 그것이 오전인지 오후인지, 어느 시간대에 발생한 데이터인지 파악할 수 없다. 데이터 분석 시 시계열 정보가 부족해진다.

**어디 부분인지**
- 파일: `SensorData.cs` (전체)
- 파일: `DataGenerator.cs` 라인 236-250 (SensorData 생성 시 SimHour 미포함)

**개선을 어떻게 하면 좋을지**
- SensorData에 SimHour 추가:
  ```csharp
  public class SensorData
  {
      public int SimHour { get; init; }  // 0~23
      public double Temperature { get; init; }
      // 기존 필드들
  }
  ```
- DataGenerator에서 생성 시 포함:
  ```csharp
  return new SensorData
  {
      SimHour = _simHour,
      Temperature = temperature,
      // 기존 필드들
  };
  ```
- CSV 헤더에 추가하여 저장

### 14. MainWindow의 매직 넘버 및 Brush 중복 생성

**어떤 문제인지**

메직 넘버로 인해 코드 의도가 명확하지 않으며 나중에 교차로 크기 변경 시 모든 숫자를 찾아 수정해야 한다. DrawGraph() 루프 내에서 매 tick마다 50개 이상의 Brush 객체가 생성되어 GC 압박을 주고 성능을 저하시킨다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 469-470, 493-496, 583-586 (좌표)
- 파일: `MainWindow.xaml.cs` 라인 306-351 (DrawGraph에서 BrushConverter 반복)

  private const int CANVAS_WIDTH = 600;

  private readonly Brush _brushGraphLine = (Brush)new BrushConverter().ConvertFrom("#64B5F6");
