### 1. DataManager 완성 및 저장 검증

**어떤 문제인지**
DataManager.cs가 생성되어 있지만 CSV/JSON 저장 기능이 완전하지 않고, AutoSaveDay() 반환값의 검증이 없음.

**어디 부분인지**
- 파일: `DataManager.cs` 전체, `MainWindow.xaml.cs` 라인 661, 674-678
- 메서드: `AutoSaveDay()`, `OnDayCompleted()`
- 코드:
  ```csharp
  bool isSaved = _dataManager.AutoSaveDay();
  // 단순 bool만 반환, 상세 에러 정보 없음
  ```

**개선을 어떻게 하면 좋을지**
- DataManager에 상세한 로깅 추가 (성공/실패 이유, 파일 경로)

### 2. 역주행 및 빙판 강제 조작 기능 수정

**어떤 문제인지**
"역주행 시뮬레이션 발생" 버튼이 명세에서는 역주행을 발생시키도록 되어 있지만, 실제로는 긴급 정지(Emergency Stop)만 작동하고 데이터상에 역주행을 발생시키지 않음. 마찬가지로 UI에서 강제로 빙판 상황을 트리거할 수 있는 수단이 없음.

**왜 문제인가?**
버튼의 이름과 실제 동작이 불일치한다. 사용자는 "역주행을 시뮬레이션"하려고 버튼을 누르지만, 실제로는 "긴급 정지만" 발동된다. DataGenerator에서 역주행과 빙판 상황은 랜덤으로만 발생하고, UI 버튼에서는 강제로 발생시킬 방법이 없다. 테스트 시 역주행/빙판 상황을 의도적으로 만들 수 없어 테스트 커버리지가 낮다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 637, `DataGenerator.cs` 라인 379-382
- 파일: `TrafficController.cs` 라인 34 부근 (변수 선언부), 라인 138-150 (`CalculateRoadSafetyScore`), 라인 160-182 (`ExecuteSignalControl`)

### 3. 타이머 동기화 (PassTime 기반 리팩토링)

**어떤 문제인지**
시뮬레이션 배속을 올려도 신호등 카운트다운 속도가 변하지 않으며, 신호 전환 시점에서 "신호 전환 대기 중..." 상태가 너무 길게 지속됨.

**왜 문제인가?**
1. `TrafficController`가 `DateTime.Now`(현실 시간)를 기준으로 초를 계산하기 때문에 배속(2x, 4x)이 적용되어도 신호등은 현실 시간 1배속으로 동작함.
2. UI는 30FPS로 즉시 0초에 도달하지만, `TrafficController`는 `DataGenerator`로부터 다음 센서 데이터(Tick)를 수신받을 때까지 신호 전환 연산을 수행하지 않음.
3. `PassTime`에서 가상 시간을 한 번 뺐는데, `UpdateNormalPhase` 내부에서 `DateTime.Now - _lastStateChangeTime`으로 현실 시간을 또 빼버리는 문제가 발생함.
4. `UpdateNormalPhase`에서 `active1`, `wait1` 같은 임시 참조 변수를 만들어 색상과 시간을 조작하려다 보니, 원본 `Light_N` 객체와의 연결이 끊어져서 UI 화면이 0초에서 정지함.

**어디 부분인지**
- 파일: `TrafficController.cs` 라인 77-83 (`PassTime`)
- 파일: `TrafficController.cs` 라인 185-232 (`UpdateNormalPhase`, `SwitchPhase`)

**개선을 어떻게 하면 좋을지**

#### 문제 분석
`DateTime`과 배속적용을 위한 `PassTime`을 함께 조화롭게 사용하기 위한 명확한 해결책이 필요함. `DateTime.Now`를 이용한 정밀한 시간 연산과 배속 계산은 UI(`MainWindow`)가 전담하여 `PassTime`으로 전달하고, `TrafficController`는 내부의 중복된 `DateTime` 연산을 제거해서 이중 차감 충돌을 방지해야 함.

또한, 복사된 값을 사용하기 위한 임시 변수 사용 대신, `isNorthSouthPhase` 플래그에 따라 방향별 원본 신호등에 직접 값을 대입해서 화면 멈춤(=참조 끊김) 현상을 해결해야 함.

#### 구현 방법

**1. PassTime 메서드 - 배속이 반영된 시간만 처리**
```csharp
public void PassTime(double deltaTime)
{
    if (_isEmergencyActive) return;

    _currentPhaseTotalTime -= deltaTime;
    UpdateNormalPhase();
}
```

**2. UpdateNormalPhase - 원본 객체에 직접 대입, DateTime 연산 제거**
```csharp
private void UpdateNormalPhase()
{
    if (_isEmergencyActive)
    {
        _lastStateChangeTime = DateTime.Now;
        _isEmergencyActive = false;
    }

    // 소수점인 남은 시간을 화면 표시용 정수로 변환
    int remainingSeconds = Math.Max(0, (int)Math.Ceiling(_currentPhaseTotalTime));

    // 참조가 끊어지지 않도록 객체에 직접 명시적 대입
    if (isNorthSouthPhase)
    {
        Light_N.RemainingTime = remainingSeconds;
        Light_S.RemainingTime = remainingSeconds;
        Light_E.Color = LightColor.Red;
        Light_W.Color = LightColor.Red;
        Light_E.RemainingTime = 0;
        Light_W.RemainingTime = 0;
    }
    else
    {
        Light_E.RemainingTime = remainingSeconds;
        Light_W.RemainingTime = remainingSeconds;
        Light_N.Color = LightColor.Red;
        Light_S.Color = LightColor.Red;
        Light_N.RemainingTime = 0;
        Light_S.RemainingTime = 0;
    }

    if (remainingSeconds <= 0)
    {
        SwitchPhase();
    }
}
```

**3. SwitchPhase - 신호 전환 직후 즉시 시간 업데이트**
```csharp
private void SwitchPhase()
{
    // ... 기존 신호 색상 전환 로직 ...

    _isExtendedThisPhase = false;

    // 신호가 바뀌자마자 바뀐 시간을 즉시 전달해서, 0초 딜레이(=무한한 "대기 중...") 현상 방지
    int updatedSeconds = (int)Math.Ceiling(_currentPhaseTotalTime);
    if (isNorthSouthPhase)
    {
        Light_N.RemainingTime = updatedSeconds;
        Light_S.RemainingTime = updatedSeconds;
    }
    else
    {
        Light_E.RemainingTime = updatedSeconds;
        Light_W.RemainingTime = updatedSeconds;
    }
}
```

### 6. TrafficLight 참조 공유로 데이터 오염

**어떤 문제인지**
TrafficUpdateEventArgs에서 TrafficLight 참조를 공유하므로 이후 TrafficController의 상태 변경이 저장된 과거 데이터에도 반영된다.

**왜 문제인가?**
TrafficLight는 클래스(참조 타입)이므로 DataManager._history 리스트의 모든 TrafficUpdateEventArgs가 동일한 Light_N 객체를 참조한다. TrafficController가 Light_N.Color를 변경하면 _history의 모든 항목(과거 24시간 데이터)의 Light_N.Color도 현재 값으로 변경된다. CSV 저장 시 모든 행의 신호가 동일하게 기록되어 데이터 무결성 오류가 발생한다.

**어디 부분인지**
- 파일: `TrafficController.cs` 라인 112-115 (`OnSensorDataUpdated`)
- 파일: `DataManager.cs` 라인 17-25 (`RecordData` 메서드)

**개선을 어떻게 하면 좋을지**

#### 전략: UI는 원본 참조, DataManager는 복사본 저장

TrafficController에서 `.Clone()` 호출을 제거하여 **UI와 원본 객체 참조를 공유**하되, 결과 데이터를 저장하는 **DataManager에서만 `.Clone()`을 사용**해서 복사본으로 안전하게 저장하여 UI 업데이트와 CSV 저장 둘 다 참조 문제를 해결함.

#### 구현 방법

**1. TrafficController - Clone() 제거하여 원본 참조 전달 (이미 구현됨)**
```csharp
TrafficUpdated?.Invoke(this, new TrafficUpdateEventArgs
{
    Timestamp = DateTime.Now,
    CurrentState = currentState,
    EfficiencyScore = efficiencyScore,
    SafetyScore = safetyScore,

    // Clone()을 제거하여 UI와 원본 객체 참조 공유
    Light_N = this.Light_N,
    Light_S = this.Light_S,
    Light_E = this.Light_E,
    Light_W = this.Light_W,

    RawData = data
});
```

**2. DataManager - RecordData에서 복사본으로 안전하게 저장**
```csharp
public void RecordData(TrafficUpdateEventArgs data)
{
    if (data == null) return;

    // UI용 원본 참조는 유지하되, CSV 저장용 리스트에는 복사본을 생성하여 저장
    var snapshot = new TrafficUpdateEventArgs
    {
        Timestamp = data.Timestamp,
        CurrentState = data.CurrentState,
        EfficiencyScore = data.EfficiencyScore,
        SafetyScore = data.SafetyScore,
        RawData = data.RawData,

        // Clone() 메서드 사용하여 참조 끊김
        Light_N = data.Light_N?.Clone(),
        Light_S = data.Light_S?.Clone(),
        Light_E = data.Light_E?.Clone(),
        Light_W = data.Light_W?.Clone()
    };

    lock (_lock)
    {
        _history.Add(snapshot); // 오염되지 않는 안전한 스냅샷
    }
}
```

**3. TrafficLight.Clone() 메서드 (이미 구현됨)**
```csharp
public TrafficLight Clone() => new TrafficLight
{
    Color = this.Color,
    RemainingTime = this.RemainingTime
};
```

이 방식으로 UI는 실시간 신호등 상태를 반영받으면서도, CSV 저장 데이터는 각 시점의 정확한 신호 상태를 기록할 수 있다.

### 7. IDisposable 미준수 및 메모리 누수

**어떤 문제인지**
MainWindow에서 리소스 정리가 불완전하고 이벤트 핸들러가 해제되지 않는다.

**왜 문제인가?**
_dataGenerator는 IDisposable을 구현하지만 Stop()만 호출되고 Dispose()는 호출되지 않는다. TrafficController는 DataGenerator 이벤트를 구독한 후 해제 수단이 없다. MainWindow의 익명 람다 이벤트 핸들러(라인 153, 172)가 정리되지 않는다. _renderTimer도 명시적 정리가 누락된다. 장기 실행 시 메모리 누수와 예측 불가능한 동작이 발생할 수 있다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 686 (Window_Closing)
- 파일: `TrafficController.cs` 라인 49-57 (생성자, 구독 해제 불가)
- 파일: `DataGenerator.cs` 라인 101-104 (Dispose 미완성)

**개선을 어떻게 하면 좋을지**
- MainWindow의 Window_Closing에서 명시적 정리:
  ```csharp
  private void Window_Closing(object sender, System.ComponentModel.CancelEventArgs e)
  {
      _renderTimer?.Stop();
      _renderTimer = null;
      _dataGenerator?.Dispose();
      _dataManager?.ClearHistory();
  }
  ```
- TrafficController에 Dispose 구현:
  ```csharp
  private DataGenerator _generator;

  public TrafficController(DataGenerator generator)
  {
      _generator = generator;
      _generator.SensorDataUpdated += OnSensorDataUpdated;
  }

  public void Dispose()
  {
      _generator.SensorDataUpdated -= OnSensorDataUpdated;
  }
  ```

### 8. 계층 분리

**어떤 문제인지**
UI(MainWindow)가 도메인 로직(TrafficController, DataManager)을 직접 처리하고 있어서 UI와 비즈니스 로직이 강하게 결합되어 있음.

**왜 문제인가?**
Score Logic이나 다른 모듈에서 신호 제어 로직을 건드려야 할 때 UI 코드를 거쳐야 하므로 변수명과 구조가 꼬일 가능성이 높다. 여기서 더 복잡해지면 유지보수가 불가능해진다. 또한 TrafficController의 정책 변경이나 성능 최적화를 테스트할 수 없으므로 버그 발견이 어렵다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 153-169
- 메서드: `InitializeDataGenerator()`의 TrafficUpdated 이벤트 핸들러
- 코드:
  ```csharp
  _trafficController.TrafficUpdated += (s, e) =>
  {
      this.Dispatcher.Invoke(() =>
      {
          _dataManager.RecordData(e);           // ← UI에서 데이터 저장
          ApplyTrafficPolicy(...);              // ← UI에서 정책 적용
      });
  };
  ```

**개선을 어떻게 하면 좋을지**
- 중간 계층 클래스 `TrafficPolicyManager` 추가
- 이 클래스가 TrafficController 이벤트를 받아서 DataManager와 UI에 필요한 데이터만 전달
- 구조:
  ```
  TrafficController → TrafficPolicyManager → MainWindow (UI만 담당)
                   ↓
                DataManager (저장 담당)
  ```
- UI는 렌더링만, 로직은 별도로 테스트 가능

### 9. 렌더링 성능 저하

**어떤 문제인지**
RenderTimer_Tick에서 40개 차량 × 모든 차량을 검사하는 O(N²) 충돌 감지 알고리즘을 사용하고 있어서 매 프레임마다 불필요한 거리 계산이 반복됨.

**왜 문제인가?**
현재 RenderTimer는 30FPS (33ms마다 Tick)로 설정되어 있는데, 40개 차량에서 O(N²) 검사를 30ms 안에 처리하려면 각 Tick마다 1,600번의 거리 계산이 발생한다. 현재는 견딜 만하지만, 차량이 추가되거나 복잡한 로직이 추가되면 프레임이 드롭되어 UI가 버벅거린다.

**어디 부분인지**
- 파일: `MainWindow.xaml.cs` 라인 502-509
- 메서드: `RenderTimer_Tick()`의 앞차 거리 조절 로직
- 코드:
  ```csharp
  for (int i = 0; i < MAX_CARS; i++)
  {
      for (int j = 0; j < MAX_CARS; j++)      // ← O(N²) 검사
      {
          if (car.Direction == "N" && other.Direction == "N" && other.Y - car.Y < 25)
          { canMove = false; break; }          // ← 거리 계산 반복
      }
  }
  ```

**개선을 어떻게 하면 좋을지**
- 각 방향별로 차량 리스트 분리:
  ```csharp
  private List<CarUI> _carsN = new List<CarUI>();
  private List<CarUI> _carsS = new List<CarUI>();
  private List<CarUI> _carsE = new List<CarUI>();
  private List<CarUI> _carsW = new List<CarUI>();

  // 검사할 때 같은 방향 차량끼리만 비교
  for (int i = 0; i < _carsN.Count; i++)
  {
  ```

### 10. 예외 처리 강화

**어떤 문제인지**

  ```csharp
  catch (Exception ex)
  ```

**어디 부분인지**
