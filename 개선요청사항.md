# 개선요청사항

## UI 렌더링 간에 신호등시간 고정 또는 점수 100점 고정 문제

**어떤 문제인지**
센서 데이터가 새로 업데이트되더라도, UI의 대시보드에서 점수가 100점으로 고정되어있거나 신호등의 남은 시간을 표현하는 게이지가 깎이지 않음.

**왜 문제인가?**
`TrafficController`에서 이벤트를 발생시킬 때 `.Clone()`을 사용해서 UI에 복사본을 전달한다. 그 뜻은, 컨트롤러 안에서 센서데이터를 새로 업데이트해도 UI에서 가지고 있는 복사본은 처음 받은 시간 그대로이고 갱신되지 않기 때문에 문제가 발생했다.

**어디 부분인지**

- 파일: `TrafficController.cs` 라인 90-93 (`OnSensorDataUpdated` 내부)

**개선을 어떻게 하면 좋을지**
`.Clone()` 호출이 아니라 원본 객체의 참조를 그대로 UI에 전달해서 컨트롤러가 시간을 깎으면 UI에 그대로 전달되어야 한다.
<br>
그런데 `TrafficLight`는 클래스라서 참조를 공유하니까, 과거기록이 계속 최신값으로 오염되는 문제가 있어서 `.Clone()`사용이 제안된거다. 그래서 이 방법을 삭제하자는 말이 아니다. 바로 뒤의 내용에서 다룬다.

```csharp
TrafficUpdated?.Invoke(this, new TrafficUpdateEventArgs
{
    Timestamp = DateTime.Now,
    CurrentState = currentState,
    EfficiencyScore = efficiencyScore,
    SafetyScore = safetyScore,

    // Clone()을 제거하여 UI와 원본 객체 참조 공유
    Light_N = this.Light_N,
    Light_S = this.Light_S,
    Light_E = this.Light_E,
    Light_W = this.Light_W,

    RawData = data
});
```

## `TrafficLight` 참조 공유로 인한 과거 데이터 오염 문제

**어떤 문제인지**
`TrafficUpdateEventArgs`에서 `TrafficLight` 참조를 그대로 공유하여 기록하기 때문에, 이후 `TrafficController`에서 신호등의 색상이나 시간이 변경되면, 이미 저장되어 있던 과거 히스토리 데이터까지 현재 상태로 전부 동기화되어 덮어씌워지는 현상이 발생함.

**왜 문제인가?**
`TrafficLight`는 클래스(참조 타입)이기 때문에 `DataManager._history` 리스트에 추가된 모든 과거 객체들이 실시간으로 변하는 컨트롤러의 원본 `Light_N` 객체 주소를 똑같이 가리키게 됨. 이로 인해 시뮬레이션이 진행되는 동안 과거 기록이 계속 최신 값으로 오염되며, 최종적으로 하루 시뮬레이션이 끝나고 CSV 파일로 저장할 때 모든 행(Row)의 신호등 상태가 맨 마지막 종료 직전의 신호 색깔로 일괄 도배되는 치명적인 데이터 무결성 오류가 발생함.

**어디 부분인지**

- 파일: `DataManager.cs` 라인 20-27 (`RecordData` 메서드 내부)

**개선을 어떻게 하면 좋을지**
`TrafficController`에서 `.Clone()`을 사용하면 UI 초시계가 멈추는 렌더링 버그가 생기니까, 원본참조를 넘겨주도록 유지하고,그 대신 결과데이터를 저장하는 `DataManager`에서 `.Clone()`을 사용해서 복사본으로 안전하게 저장하면 UI와 CSV 저장 둘 다 참조 문제를 해결할 수 있다.

```csharp
public void RecordData(TrafficUpdateEventArgs data)
{
    if (data == null) return;

    // UI용 원본 참조는 유지하되, CSV 저장용 리스트에는 복사본을 생성하여 저장
    var snapshot = new TrafficUpdateEventArgs
    {
        Timestamp = data.Timestamp,
        CurrentState = data.CurrentState,
        EfficiencyScore = data.EfficiencyScore,
        SafetyScore = data.SafetyScore,
        RawData = data.RawData,

        // Clone() 메서드 사용
        Light_N = data.Light_N?.Clone(),
        Light_S = data.Light_S?.Clone(),
        Light_E = data.Light_E?.Clone(),
        Light_W = data.Light_W?.Clone()
    };

    lock (_lock)
    {
        _history.Add(snapshot); // 오염되지 않는 안전한 스냅샷
    }
}
```

## 수동 강제조작에 대해 동일하게 패널티 부여

**어떤 문제인지**
가장 처음 개선 제안과 관련있다. 수동제어 버튼을 눌렀을 때 강제된 상황이 `TrafficController`에 반영되도록 제안하면서, 수동으로 조작된 상황에 대해서도 동일하게 시나리오가 적용되도록 변경할 필요가 있다.

**왜 문제인가?**
`CalculateRoadSafetyScore` 로직이 오직 랜덤하게 생성된 SensorData의 문자열(건조/습윤/결빙) 상태만 순수하게 검사하고 있기 때문에, UI 버튼으로 트리거되는 예외 상황을 알 방법이 없다.

**어디 부분인지**

- 파일: `TrafficController.cs` 라인 34 부근 (변수 선언부), 라인 115-127 (`CalculateRoadSafetyScore`), 라인 137-159 (`ExecuteSignalControl`)

**개선을 어떻게 하면 좋을지**
외부(UI)에서 강제된 상태를 통보받을 수 있는 `public bool` 플래그를 변수 선언부에 추가하고, 점수를 계산하거나 신호를 연장할 때 해당 플래그를 조건문에 포함시켜 강제 제어 시나리오를 정상적으로 인식하도록 수정을 제안한다.

```csharp
public bool ForceBlackIce { get; set; } = false;
public bool ForceWrongWay { get; set; } = false;
```

```csharp
private int CalculateRoadSafetyScore(SensorData data)
{
    int score = SAFETY_BASE_SCORE;

    // 강제 빙판 모드일 때도 인식 (ForceBlackIce 조건 추가)
    if (ForceBlackIce || data.RoadCondition == "결빙" || data.RoadCondition == "적설") score += FREEZING_SNOW_PENALTY;
    else if (data.RoadCondition == "습윤") score += WET_PENALTY;

    if (data.IsPedestrianRemaining) score += PEDESTRIAN_PENALTY;

    // 강제 역주행 모드일 떄도 인식 (ForceWrongWay 조건 추가)
    if (ForceWrongWay || data.IsWrongWay) score += WRONG_WAY_PENALTY;

    return score;
}
```

## `TrafficControl.cs`에서의 `PassTime` 도입으로 인한 일부 코드 수정 제안

**어떤 문제인지**
UI 프레임 배속을 맞추기 위해 `PassTime(double deltaTime)`을 도입했으나, 신호가 0초에 도달한 뒤 다음 신호로 넘어가지 못하고 "신호 전환 대기 중..." 상태에서 영원히 멈춘다.

**왜 문제인가?**

- `PassTime`에서 가상 시간을 한 번 뺐는데, `UpdateNormalPhase` 내부에서 `DateTime.Now - _lastStateChangeTime`으로 현실 시간을 또 빼버리는 문제가 발생함.

- `UpdateNormalPhase`에서 `active1`, `wait1` 처럼 임시 참조 변수를 만들어 색상과 시간을 조작하려다 보니, 원본 `Light_N` 객체와의 연결이 끊어져서 UI 화면이 0초에서 정지함.

**어디 부분인지**

- 파일: `TrafficController.cs` 라인 162-188 (`UpdateNormalPhase`), 라인 190-210 (`SwitchPhase`)

**개선을 어떻게 하면 좋을지**
랙 방지를 위해 제안된 `DateTime`과 배속적용을 위한 함수 `PassTime`을 함께 조화롭게 사용하기 위한 명확한 해결책을 제안한다. `DateTime.Now`를 이용한 정밀한 시간 연산과 배속 계산은 UI(`MainWindow`)가 전담하여 `PassTime`으로 전달하고, `TrafficController`는 내부의 중복된 `DateTime` 연산을 제거해서 이중 차감 충돌을 방지할 것을 제안함.
또한, 복사된 값을 사용하기 위한 임시 변수 사용 대신, `isNorthSouthPhase` 플래그에 따라 방향별 원본 신호등에 직접 값을 대입해서 화면 멈춤(=참조 끊김) 현상을 해결하도록 제안한다.

```csharp
private void UpdateNormalPhase()
{
    if (_isEmergencyActive) _isEmergencyActive = false;

    // 소수점인 남은 시간을 화면 표시용 정수로 변환
    int remainingSeconds = Math.Max(0, (int)Math.Ceiling(_currentPhaseTotalTime));

    // 참조가 끊어지지 않도록 객체에 직접 명시적 대입
    if (isNorthSouthPhase)
    {
        Light_N.RemainingTime = remainingSeconds; Light_S.RemainingTime = remainingSeconds;
        Light_E.Color = LightColor.Red; Light_W.Color = LightColor.Red;
        Light_E.RemainingTime = 0; Light_W.RemainingTime = 0;
    }
    else
    {
        Light_E.RemainingTime = remainingSeconds; Light_W.RemainingTime = remainingSeconds;
        Light_N.Color = LightColor.Red; Light_S.Color = LightColor.Red;
        Light_N.RemainingTime = 0; Light_S.RemainingTime = 0;
    }

    if (remainingSeconds <= 0) SwitchPhase();
}
```

```csharp
private void SwitchPhase()
{
    // ... 기존 코드 ...

    _isExtendedThisPhase = false;

    // 신호가 바뀌자마자 바뀐 시간을 즉시 전달해서, 0초 딜레이(=무한한 "대기 중...") 현상 방지
    int updatedSeconds = (int)Math.Ceiling(_currentPhaseTotalTime);
    if (isNorthSouthPhase) {
        Light_N.RemainingTime = updatedSeconds;
        Light_S.RemainingTime = updatedSeconds;
    }
    else {
        Light_E.RemainingTime = updatedSeconds;
        Light_W.RemainingTime = updatedSeconds;
    }
}
```
